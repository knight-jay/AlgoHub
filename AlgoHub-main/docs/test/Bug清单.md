# Bug清单
记录系统测试过程中发现的问题及优化点，状态实时更新。

============================================================
# 一、功能缺陷（Bug）
============================================================

## 1. 修改密码：新旧密码一致仍可修改成功
- 所属模块：用户模块 - 修改密码
- 问题描述：旧密码、新密码完全相同时，依然可以修改成功
- 预期结果：提示“新密码不能与旧密码相同”，拒绝修改
- 实际结果：修改成功
- 建议：新增新旧密码对比逻辑，一致则不更新
- 状态：已修复

## 2. 修改个人信息：信息未变化仍提示修改成功
- 所属模块：用户模块 - 修改个人信息
- 问题描述：手机号、昵称等信息前后一致，提交后仍提示操作成功
- 预期结果：数据无变化时，提示“信息未发生修改”
- 实际结果：始终提示“修改成功”
- 建议：后端对比新旧数据，无变化不执行更新
- 状态：已修复

## 3. 用户注册：昵称（nickname）为空可注册成功
- 所属模块：用户模块 - 注册
- 问题描述：注册时 nickname 传入空值/空字符串，仍能注册成功
- 预期结果：昵称不能为空，应提示“昵称不能为空”
- 实际结果：可以正常注册
- 建议：后端增加非空校验，nickname 不允许为空
- 状态：已修复

## 4. 管理员操作：允许禁用自身账号（自杀漏洞）
- 所属模块：管理员模块 - 禁用/启用账号
- 问题描述：管理员使用自己的账号调用禁用接口，传入自己的 User ID 时，未被拦截
- 预期结果：拦截操作并提示“不能对自身账号进行禁用操作”
- 实际结果：操作成功，导致当前管理员账号被禁用（无法再次登录或管理）
- 建议：后端从 token 中解析当前登录用户 ID，若目标 ID 与当前 ID 一致，则拦截报错
- 状态：已修复

## 5. 管理员操作：允许降低自身权限
- 所属模块：管理员模块 - 修改角色
- 问题描述：管理员操作自己的账号，将角色改为 STUDENT，操作未被拦截
- 预期结果：拦截操作并提示“禁止降低自身管理权限”
- 实际结果：操作成功，管理员立刻丧失管理权限
- 建议：与 Bug-4 类似，判断操作对象是否为本人，阻止敏感权限降级操作
- 状态：已修复

## 6. 算法搜索：keyword 传入纯空格未被拦截
- 所属模块：算法知识库 - 搜索算法
- 问题描述：调用搜索接口传入纯空格作为关键字，后端未做去空及非空校验，直接带入数据库执行模糊查询
- 预期结果：纯空格视为无效搜索内容，应拦截并提示“搜索关键字不能为空”
- 实际结果：未做任何拦截，直接触发全表数据查询，返回所有算法数据，存在全表扫描性能隐患
- 建议：后端对关键字先执行首尾去空处理，校验处理后为空字符串则直接拦截返回友好提示
- 状态：已修复

## 7. 学习资源搜索：代码拦截逻辑失效导致纯空格查全量
- 所属模块：学习资源（公开）模块 - 搜索资源 (`/api/resources/search`)
- 问题描述：输入连续空格测试时，后端代码中的 `if (keyword.trim().isEmpty() && !keyword.isEmpty())` 校验拦截失效
- 预期结果：触发条件，返回报错“搜索关键字不能为空”。
- 实际结果：条件判断未生效，直接返回了 code: 200 并触发了默认的全量分页查询返回所有资源数据。
- 建议：排查 Spring `@RequestParam` 传参时是否自动抹除了空白字符导致 `!keyword.isEmpty()` 判定为 false
- 状态：已修复

## 8. 帖子搜索：keyword 传入纯空格未被拦截
- 所属模块：问答社区 - 搜索帖子（`/api/posts/search`）
- 问题描述：调用搜索接口传入纯空格作为关键字，后端未做 trim 判空校验，直接带入数据库模糊查询
- 预期结果：纯空格视为无效搜索内容，应拦截并提示"搜索关键字不能为空"
- 实际结果：未做任何拦截，直接返回全表帖子数据
- 建议：后端对 keyword 先执行 trim，判空则直接拦截返回友好提示
- 状态：已修复

## 9. 点赞/收藏/关注用户：toggle 操作未校验目标是否存在及合法性
- 所属模块：问答社区 - 点赞帖子（`POST /api/posts/{id}/like`）、关注用户（`POST /api/users/{id}/follow`）
- 问题描述：
  1. 对不存在的帖子 ID（如 99999）调用点赞接口，后端未校验帖子是否存在，返回"已取消点赞"
  2. 用户关注自己时，后端未校验，返回"已取消关注"
  3. 收藏（favorite）、关注帖子（follow）同样存在目标不存在时返回误导结果的问题
- 预期结果：应先判断目标是否存在及操作合法性，不存在返回"帖子/用户不存在"，自己关注自己应返回"不能关注自己"
- 实际结果：code: 200，返回"已取消点赞/已取消关注"等误导性提示
- 建议：toggle 方法开头先校验目标存在性和操作合法性，不合法直接返回对应错误提示
- 状态：已修复

## 10. 删除帖子/评论：非作者调用返回 success 但实际未删除
- 所属模块：问答社区 - 删除帖子（`DELETE /api/posts/{id}`）、删除评论（`DELETE /api/comments/{id}`）
- 问题描述：非作者用户调用删除接口，service 层校验不通过后静默 return，但 controller 层始终返回 "删除成功"
- 预期结果：非作者删除时返回错误提示"无权删除"或"帖子不存在或无权编辑"
- 实际结果：code: 200，msg: "删除成功"，但数据库中数据未被删除
- 建议：将 `deletePost` 和 `deleteComment` 方法改为返回 boolean 或抛异常，controller 根据返回值决定返回成功还是错误提示
- 状态：已修复

# 二、待优化建议（规范/体验优化）
============================================================

## 1. 用户名允许中间包含空格
- 所属模块：用户模块 - 注册
- 当前测试发现：
  1. 纯空格不允许注册
  2. 中间空格允许，且计入总长度
  3. 长度 3~20 之间可正常注册
  4. 超出长度则拒绝
- 优化说明：用户名属于账号，正规系统通常不允许包含空格
- 建议：禁止用户名包含任何空格（前后、中间均不允许）
- 状态：已优化

## 2. 用户列表查询：缺乏分页机制可能导致内存溢出（非必要优化）
- 所属模块：管理员模块 - 获取用户列表
- 问题描述：`listUsers` 接口直接调用 `findAll()` 查出全表数据，没有分页设计
- 预期结果：应支持传入 `page` 和 `size` 参数，按页返回数据
- 实际结果：返回数据库中所有用户，若用户量达到数万，将导致系统严重卡顿或 OOM
- 建议：后端引入 Spring Data JPA 的 `Pageable` 分页对象改造查询
- 状态：已优化

## 3. 分页接口未限制 pageSize 最大值，存在内存溢出风险
- 所属模块：算法知识库 - 所有分页相关接口
- 问题描述：分页接口未限制单次查询最大条数，可随意传入 `pageSize=100000` 超大数值
- 预期结果：限制单次最大查询条数，如最大不超过100条，防止恶意批量拉取数据
- 实际结果：无上限限制，一次性可查出巨量数据，极易造成接口卡顿、数据库压力过大、内存溢出OOM
- 建议：给 pageSize 增加最大值限制，或业务层强制兜底截断
- 状态：已优化

## 4. 查询不存在分类仅返回空列表，无友好文字提示
- 所属模块：算法知识库 - 按分类查询算法
- 问题描述：传入不存在分类ID如99999，接口只返回200空列表，无任何文字说明
- 预期结果：应友好提示“该分类不存在或已删除”
- 实际结果：仅返回空数据，用户无法区分是「分类不存在」还是「分类下暂无数据」
- 建议：后端先校验分类ID是否存在，不存在则返回对应错误提示
- 状态：已优化
