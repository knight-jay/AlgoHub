## 1. 测试目标

- 验证算法管理（管理员）模块 3 个功能（新增算法、修改算法、删除算法）的正确性，确保管理员可正常管理算法数据。
- 验证参数校验：修改不存在的算法 ID 时返回错误提示。

## 2. 测试范围

| 模块编号 | 功能模块 | 接口方法及路径 | 测试用例数 |
|---|---|---|---|
| M1 | 新增算法 | `POST /api/admin/algorithm` | 1 |
| M2 | 修改算法 | `PUT /api/admin/algorithm/{id}` | 2 |
| M3 | 删除算法 | `DELETE /api/admin/algorithm/{id}` | 1 |
| **合计** | | | **4** |

## 3. 测试策略

### 3.1 测试类型
接口功能测试（黑盒），通过 Apifox 发送 HTTP 请求，校验 HTTP 状态码及响应 JSON 体。需携带管理员 Token。

### 3.2 用例设计方法
- **正向用例：** 管理员 Token + 合法参数，验证返回 `code: 200` 且操作生效（新增、修改、删除）。
- **异常用例——资源校验：** 修改不存在的算法 ID（99999），验证返回 `code: 500`，"算法不存在"。

### 3.3 测试流程
1. 管理员登录获取 Token。
2. 先执行新增算法，为后续修改/删除提供测试数据。
3. 执行修改用例（正常修改 + 不存在 ID）。
4. 执行删除用例，验证删除生效。

## 6. 通过/失败标准

**通过标准：**
- 管理员正向操作返回 `code: 200`，新增/修改/删除操作生效。
- 修改不存在算法返回 `code: 500`，"算法不存在"。

**失败标准：**
- 正向操作未生效或返回非 200。
- 修改不存在算法未拦截，返回成功提示但数据未变更。

## 7. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 删除算法后关联数据未处理 | 引用该算法的收藏/评论等数据悬空 | 建议后续补充关联数据级联处理逻辑 |
| 新增/修改未做字段校验 | 非法数据入库 | 后续补充 title/content 空值、difficulty 非法值等校验用例 |
