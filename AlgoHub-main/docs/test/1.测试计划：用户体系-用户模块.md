## 1. 测试目标

- 验证用户模块 5 个功能（注册、登录、查看个人信息、修改个人信息、修改密码）的正确性，确保核心用户流程可用。
- 验证参数校验是否生效：用户名重复、两次密码不一致、角色不匹配、密码错误等场景能被正确拦截。
- 验证鉴权控制：未携带 Token 访问需登录接口时返回 401。
- 发现并记录功能缺陷，输出至 Bug 清单，跟踪修复状态。

## 3. 测试策略

### 3.1 测试类型
接口功能测试（黑盒），通过 Apifox 发送 HTTP 请求，校验 HTTP 状态码及响应 JSON 体。

### 3.2 用例设计方法
- **正向用例：** 合法输入，验证返回 `code: 200` 且业务数据正确（注册成功、登录获取 Token、信息修改生效）。
- **异常用例——参数校验：** 用户名重复、两次密码不一致、角色不匹配、密码错误，验证返回 `code: 500` 并携带明确错误提示。
- **异常用例——鉴权校验：** 不带 Token 访问需登录接口，验证返回 `code: 401`。
- **异常用例——业务规则校验：** 新旧密码相同、用户名含空格，验证后端业务校验逻辑生效。

### 3.3 测试流程
1. 按功能模块顺序执行：注册 → 登录 → 查看个人信息 → 修改个人信息 → 修改密码。
2. 注册阶段创建测试账号（学生 test001、管理员 admin1），后续用例依赖此账号。
3. 登录阶段获取 Token，用于后续需鉴权的接口。
4. 不通过用例标注并同步追加至 Bug 清单，Bug 修复后回归验证。

## 6. 通过/失败标准

**通过标准：**
- 所有正向用例返回 `code: 200`，响应数据结构与接口文档一致。
- 所有参数校验用例正确返回 `code: 500` 并携带可读的错误提示（`msg` 字段）。
- 所有鉴权校验用例正确返回 `code: 401`。

**失败标准：**
- 正向用例未返回 200 或数据缺失/错误。
- 异常输入未被拦截（如非法用户名注册成功、错误密码登录成功）。
- 未登录访问需鉴权接口未被拦截。

## 7. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 注册账号数据残留影响重复测试 | 用户名重复用例无法复现 | 每次测试前清理或更换用户名 |
| Token 过期 | 鉴权用例误报为 401 | 测试开始前重新登录，确保 Token 有效期充足 |
| 密码规则校验遗漏 | 弱密码可注册 | 后续补充密码强度校验用例 |
| 注册/登录接口无验证码防护 | 存在暴力破解风险 | 记录为优化建议，后续版本增加验证码或限流 |
