## 一、项目概述

### 1. 项目背景

项目来源： 在AI快速发展的背景下，简单的编程任务逐渐被自动化工具替代。为了提升大学生的核心竞争力，我们小组决定开发一个算法学习平台，帮助同学们提升编码思维与算法能力，从容应对技术变革。

应用场景： 在校大学生的算法学习、竞赛备赛、校招刷题。

现实意义： 培养算法思维，提高编程实战能力，为未来就业夯实基础。

### 2. 项目目标

优先完成用户体系、算法知识库、学习资源推荐、问答社区四大核心模块，确保平台具备基本的用户管理和学习交流能力。其余模块（题解核心、题单体系、个人成长体系）根据开发进度选择性实现，作为后期拓展内容。

### 3. 开发环境与条件

开发工具： 前端使用 Vue3 + Vite + Element Plus，后端使用 Java + SpringBoot + MyBatis-Plus，数据库采用 MySQL 8.0，接口调试采用 Apifox，版本控制使用 Git + Gitee。

技术说明： 后端基于 SpringBoot 框架，通过 MyBatis-Plus 简化数据库操作，使用 RESTful API 风格与前端通信。整体技术栈在高校课程设计中常见，学习成本较低，适合小组协作开发。

## 二、需求分析

### 1. 目标用户与使用场景

目标用户为在校计算机相关专业大学生。使用场景覆盖日常课程学习、算法竞赛训练、实习招聘刷题等。用户通过浏览器访问平台，完成算法知识查阅、资源获取、社区问答等学习活动。

### 2. 核心功能清单

- 用户体系：注册、登录、找回密码、个人信息管理、角色权限（学生管理员）。
- 算法知识库：树形分类目录、算法详情展示、多维度检索、代码块一键复制、知识点关联跳转。
- 学习资源推荐：推荐优质网课刷题网站，支持分类筛选与详情展示。
- 问答社区：发布问题回答评论，富文本编辑器（支持代码块、LaTeX公式），悬赏积分，采纳回答，点赞关注及内容举报。

扩展功能（后期有余力再实现）：题解核心模块、题单体系、个人成长体系（积分任务、雷达图等）。

### 3. 非功能需求

- 性能要求：页面首屏加载时间小于2秒，API接口响应时间不超过300ms（95%请求），支持至少100人同时在线并发访问；数据库复杂查询通过索引优化，保证知识库检索流畅。
- 安全要求：用户密码使用BCrypt加密存储，前后端接口使用JWT令牌鉴权，防止越权操作；对用户输入进行XSS过滤与SQL防注入处理；敏感操作需二次验证。
- 兼容性要求：支持主流现代浏览器（Chrome 90+，Firefox 88+，Edge 90+），响应式布局适配笔记本、台式机及平板设备，移动端保证基础浏览体验。

## 三、技术路线

### 1. 总体架构

本项目采用前后端分离架构，前端负责界面展示与用户交互，后端负责业务逻辑与数据存储。前端通过 Axios 调用后端 API，后端使用 SpringBoot 框架统一处理请求，数据库使用 MySQL 保存数据。

- 前端展示层：Vue3 + Vite + Element Plus，负责页面渲染、组件交互、表单验证等。
- 后端服务层：SpringBoot + MyBatis-Plus，提供 RESTful API，实现用户认证、数据增删改查、权限控制等。
- 数据层：MySQL 8.0，存储用户、文章、问答等结构化数据。

### 2. 主要技术

- 前端技术：Vue3 (Composition API)、Vite 构建工具、Element Plus 组件库、Axios 网络请求、Vue Router 路由管理、Pinia 状态管理、富文本编辑器（支持代码块与LaTeX）。
- 后端技术：SpringBoot 2.7+、Spring Security + JWT 身份认证、MyBatis-Plus 持久层框架、MySQL Connector、HikariCP 连接池、Maven 项目管理。
- 数据库与工具：MySQL 8.0、Redis（会话缓存热点数据）、Apifox 接口文档与调试、Git + Gitee 代码托管、Lombok 简化开发。

### 3. 数据设计初步考虑

主要实体对象： 用户、角色权限、算法分类、算法知识库、学习资源、问答问题、问答回答、用户行为（点赞收藏）。

设计原则： 遵循数据库三范式，保证数据一致性；对常用查询字段建立索引，提升访问速度。初期设计8-10张核心表，覆盖四大必做模块。关键数据表包括：用户表(user)、算法文章表(article)、资源表(resource)、问题表(question)、回答表(answer)、评论表(comment)等，并通过外键或逻辑关联实现数据完整性。

### 1. Git分支与协作规则（简单清晰）

- 主分支 main：存放稳定可运行的版本，只有经过测试的代码才能合并到此分支。
- 开发分支 dev：日常开发集成分支，所有功能分支都从 dev 拉出，完成后合并回 dev。
- 功能分支 feature姓名-模块名：例如 featurechangyujie-user，每个人在自己分支上开发，避免互相干扰。
- 提交规范：每次提交写明做了什么，格式 [模块] 具体内容，如 [用户] 完成注册接口开发。
- 合并流程：每天1800前，各成员将自己的功能分支推送到远程，并发起 Pull Request 到 dev 分支，由常宇杰负责审核合并。每天开始工作前先执行 git pull origin dev 同步最新代码。
