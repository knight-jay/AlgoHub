二、分维度问题与解决方式
# 1. 用户模块遗留缺陷
1. 问题：修改密码业务逻辑存在漏洞——新旧密码完全相同时仍提示修改成功，未进行一致性校验
解决措施：后端新增新旧密码对比判断，若新旧密码一致则直接拒绝并提示"新密码不能与旧密码相同"
2. 问题：修改个人信息业务逻辑存在漏洞——信息未发生任何变化时仍提示修改成功，执行了无效的数据库写入
解决措施：后端在更新前对比新旧数据，所有字段均无变化时不执行UPDATE，提示"信息未发生修改"
3. 问题：用户注册校验不完整——昵称（nickname）传入空值或空字符串时仍可正常注册成功，缺少非空校验
解决措施：后端增加 nickname 非空校验，提交空字符串时拦截并提示"昵称不能为空"
4. 问题：用户名注册校验存在旁路——用户名中间包含空格未被拒绝，空格计入总长度后仍可正常注册
解决措施：注册校验规则中增加空格检测，用户名任意位置（前中后）包含空格均直接拒绝

# 2. 管理员模块安全漏洞
5. 问题：管理员可对自身账号执行禁用操作（自杀漏洞）——传入自身 User ID 调用禁用接口时未被拦截，操作成功后管理员账号被禁用，完全丧失管理能力
解决措施：后端从 JWT Token 中解析当前登录用户的 User ID，若操作目标 ID 与当前 ID 一致则拦截报错"不能对自身账号进行禁用操作"
6. 问题：管理员可降低自身权限（权限降级漏洞）——管理员操作自身账号将角色改为 STUDENT 时未被拦截，操作成功后立即丧失所有管理权限
解决措施：与问题5同类处理，判断操作对象是否为本人，阻止敏感权限变更操作，提示"禁止降低自身管理权限"
7. 问题：管理员角色体系中缺少超级管理员（群主）层级保护——所有管理员平级，无法形成权限阶梯，最高的管理账号也可能被其他管理员操作
解决措施：新增"群主"角色概念，群主角色不可被其他管理员修改或禁用，建立权限保护层级

# 3. 算法知识库模块问题
8. 问题：搜索接口对纯空格关键字缺少拦截——keyword 传入纯空格时后端未做去空和非空校验，直接带入数据库 LIKE 查询，导致全表扫描，存在性能隐患
解决措施：后端对 keyword 先执行首尾去空（trim），再判断处理后的字符串是否为空，为空则直接返回提示"搜索关键字不能为空"，不执行数据库查询
9. 问题：按不存在的分类ID查询时仅返回空列表，无任何友好提示——传入分类ID=99999等不存在值时接口返回200空数据，用户无法区分是"分类不存在"还是"该分类下暂无算法"
解决措施：后端先校验传入的分类ID是否存在，不存在则返回对应错误提示"该分类不存在或已删除"
10. 问题：分页接口未限制单次查询最大条数——可随意传入 pageSize=100000 等超大数值，存在恶意批量拉取数据和内存溢出风险
解决措施：对所有分页接口增加 pageSize 最大值限制（最大不超过100），业务层强制兜底截断

# 4. 论坛社区模块设计讨论
11. 问题：论坛社区数据库表结构设计需求——需要定义帖子表（post）、评论表（comment）、点赞表（post_like）、关注表（post_follow）、收藏表（post_favorite）、举报表（post_report）、用户关注表（user_follow）等多张关联表
解决措施：会议中确认了全部表结构设计，明确字段类型、约束条件、外键关系及索引策略，会后由后端立即编写 SQL 初始化脚本
12. 问题：论坛社区后端接口设计需求——需要覆盖帖子CRUD、评论CRUD、点赞/取消点赞、关注/取消关注、收藏/取消收藏、举报处理、管理员处置等完整功能链
解决措施：会议中逐一确认接口路径、请求参数、响应格式，会后在 API 文档中以 Markdown 形式统一维护
13. 问题：帖子详情页操作较多（点赞、关注、收藏、举报、评论、编辑、删除），操作入口和状态展示容易造成界面混乱
解决措施：前端按操作类型分组布局——互动区（点赞/关注/收藏）、内容区（评论编辑发表）、管理区（编辑/删除/举报），各组视觉独立，互不干扰

# 5. 前端联调问题
14. 问题：算法分类页面无法连接到后端数据——前端发起请求后未获取到分类树数据，页面空白
解决措施：排查发现接口路径拼写错误和参数传递格式不匹配，修正后恢复正常
15. 问题：首页错误处理逻辑不完善——接口请求失败时仅显示空白区域，无任何错误提示和状态反馈，用户体验差
解决措施：添加错误状态提示组件和控制台日志输出，接口失败时显示"加载失败，请稍后重试"并提供重试按钮，同时输出详细错误信息便于开发定位

# 6. 测试覆盖不足
16. 问题：测试范围仍局限于用户模块和管理员模块，算法知识库和学习资源模块的测试尚未开展
解决措施：会后立即对算法知识库模块进行接口功能测试，同步编写测试记录；学习资源模块测试在功能稳定后跟进
17. 问题：测试发现的Bug缺乏统一跟踪管理，状态更新不及时
解决措施：建立统一的 Bug清单.md 文件在仓库中维护，所有Bug统一编号、记录复现步骤与状态，测试与开发人员均可查阅和更新
18. 问题：测试结果缺乏可视化证据，仅凭文字记录说服力不足
解决措施：测试时同步截图保存，将图片统一存放在 test/images/ 目录下，测试记录中使用相对路径引用

# 7. 安全整改计划
19. 问题：当前密码存储方式为 MD5 哈希，安全性不足——MD5已被证明存在碰撞漏洞，且加盐机制未统一实现
解决措施：制定 BCrypt 迁移计划——先编写密码加密工具类，再编写迁移脚本批量更新已有用户密码，最后修改注册和登录流程中的加密/校验逻辑
20. 问题：系统缺少 XSS 跨站脚本攻击防护——用户输入的文本（帖子内容、评论、个人简介等）未经过滤直接存储和展示
解决措施：后端对所有用户输入参数进行 XSS 过滤处理，对 HTML 特殊字符进行转义，社区内容仅允许白名单标签
21. 问题：部分接口存在 SQL 拼接查询，未完全使用 JPA 参数绑定——存在 SQL 注入风险
解决措施：全面排查所有数据查询代码，将拼接 SQL 全部改为 JPA 参数绑定方式，从框架层面杜绝 SQL 注入
