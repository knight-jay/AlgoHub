二、分维度问题与解决方式
# 1. 论坛社区模块——后端已交付，前端集中推进
1. 问题：论坛社区数据库涉及7张新表（post、comment、post_like、post_follow、post_favorite、post_report、user_follow），表间关联关系复杂，外键设计需兼顾查询效率和数据一致性
解决措施：会议确认全部表结构设计——post表采用category_id分类关联、comment表采用post_id+user_id双向关联；点赞/关注/收藏表采用(user_id, target_id)联合唯一索引防止重复操作；post_report表存储举报类型和状态枚举；user_follow表记录用户间关注关系
对应提交：ed3279c（27文件，+1192行）
2. 问题：论坛社区后端接口需要同时面向普通用户和管理员提供两套API，公共接口涉及认证拦截配置
解决措施：社区公开接口（浏览帖子、查看评论）加入JWT拦截器白名单，管理员接口（删除帖子、处理举报）通过role=ADMIN校验，在WebConfig中统一配置拦截路径
3. 问题：社区模块前端缺乏TypeScript类型约束——Post、Comment、PostReport等实体在前端没有对应的类型定义，API调用缺乏类型安全
解决措施：在前端types/index.ts中一次性补充所有社区和资源模块的TypeScript接口定义，包括Post、Comment、CreatePostDTO、PostReport、Resource等完整类型
对应提交：67570cc（+70行）
4. 问题：社区模块前端API层为零——没有封装与后端接口对应的请求函数，页面组件直接裸调axios
解决措施：新建api/post.ts封装全部社区接口（帖子CRUD、评论CRUD、点赞、关注、收藏、举报、用户关注），新建api/resource.ts封装资源模块接口，统一错误处理和Token附加
对应提交：059eac7（2文件，+117行）
5. 问题：社区模块三个前台页面需从零开发——帖子列表页（Community）、帖子详情页（PostDetail）、学习资源页（Resources），交互逻辑复杂
解决措施：Community.tsx实现帖子列表（按时间/热度排序、分页）、发布帖子入口；PostDetail.tsx实现完整帖子详情（内容展示、评论列表、点赞/关注/收藏/举报操作按钮）；Resources.tsx实现资源卡片展示（分类筛选、分页）
对应提交：8f04950（3文件，+581行）
6. 问题：后台管理缺少举报管理和资源管理两个独立页面——管理员无法查看和处理用户举报，也无法对学习资源进行增删改
解决措施：新建ReportManagement.tsx——举报列表展示（举报类型、被举报内容、举报人、状态），支持"删除内容"和"驳回举报"两种处置；新建ResourceManagement.tsx——资源列表+新增/编辑/删除弹窗，分类管理子模块。同步更新App.tsx路由和AdminLayout导航菜单
对应提交：c7d6992（5文件，+472行）
7. 问题：个人信息页缺少"我的帖子"管理入口——用户发布帖子后无法在个人中心集中查看和管理自己的帖子
解决措施：在Profile.tsx中新增"我的帖子"栏目，调用postApi.myPosts获取当前用户的帖子列表，每条帖子支持编辑（弹窗修改标题/内容）和删除操作（含确认提示），列表支持分页
对应提交：a3803e6（+186行）
8. 问题：帖子详情页只有帖子作者可见操作区，但缺少编辑按钮——作者想修改已发布帖子时找不到入口
解决措施：在PostDetail.tsx中帖子作者可见的操作区补充"编辑"按钮，点击弹出编辑弹窗，调用postApi.update进行修改

# 2. 学习资源模块——后端已稳定，前端首次对接
9. 问题：学习资源模块前后端首次联调，接口路径和参数格式未完全对齐
解决措施：逐接口核对——GET /api/resources/categories（分类列表）、GET /api/resources/search（关键词搜索+分页）、GET /api/resources/category/{id}（按分类筛选）、GET /api/resources/{id}（详情），确认所有请求参数名和数据返回格式与前端API封装一致
10. 问题：学习资源搜索接口存在与算法搜索相同的纯空格拦截漏洞——keyword传入连续空格时后端校验逻辑失效，直接返回全量数据
解决措施：与算法模块同类问题一并处理，统一修改后端校验逻辑，在Service层对keyword执行trim后判断是否为空，拦截后返回友好提示
对应Bug编号：Bug-7，状态：已记录待修复

# 3. 新发现Bug与问题
11. 问题：学习资源推荐-公开模块接口测试发现1个拦截失效——纯空格搜索未被拦截（TC-R04），与算法知识库搜索Bug（Bug-6）为同类问题，根源在于Spring @RequestParam自动去空白字符导致后端校验条件失效
解决措施：统一排查所有搜索类接口的关键字校验逻辑，将.trim().isEmpty()逻辑替换为在Controller层手动获取原始参数字符串进行判断

解决措施：前端管理页面开发完成后（c7d6992），立即安排对前端交互流程的验收测试
13. 问题：Bug清单中仍有2项未修复——纯空格搜索拦截（Bug-6算法模块、Bug-7资源模块）原因相同但修复不及时
解决措施：安排章恒福在5月9日统一修复所有搜索类接口的空格校验逻辑
14. 问题：第二次会议决议中的管理员模块安全漏洞修复已提交代码，但测试人员尚未对修复进行回归验证

# 4. 社区论坛模块联调中新暴露的问题
15. 问题：论坛帖子列表页与帖子详情页之间的路由参数传递不规范——从列表跳转详情时postId通过URL params传递，但PostDetail组件中取参方式与路由定义不完全匹配
解决措施：统一使用React Router v6的useParams() hook获取路由参数，确保/post/:id与组件取参一致
16. 问题：举报管理后台页面的状态展示和处置操作交互不清晰——管理员无法直观区分"待处理"和"已处理"的举报
解决措施：举报列表按status字段分组展示，待处理举报高亮标记，已处理举报灰度显示；处置按钮根据当前状态动态变化（待处理→删除/驳回，已处理→仅查看）
