会议日期：2026年5月6号
参会人员：全体成员
记录人：常宇杰
一、会议议程
1. 项目各维度问题梳理
2. 问题解决方案讨论与确认
3. 后续行动计划安排
二、分维度问题与解决方式
1. 架构/目录结构
1. 问题：项目初期所有文件扁平存放，缺乏分层管理
解决措施：通过多轮重构逐步建立分层目录结构，明确不同类型文件的存放路径
2. 问题：后端代码包名不规范，未形成标准的包结构
解决措施：统一调整包名至规范格式，手动迁移对应文件并修正所有package声明
3. 问题：重复构建项目脚手架，产生多套后端目录，造成代码混乱
解决措施：采用“删除重建”策略，清理重复的脚手架代码，统一核心代码存放路径
4. 问题：包名调整后，相关文件未同步迁移，导致引用错误
解决措施：逐一核对包名与文件路径，完成所有文件的迁移和引用修正
5. 问题：测试类存放路径不符合行业标准，与业务代码混放
解决措施：将所有测试类统一迁移至标准的测试目录结构下
6. 问题：整体目录结构混乱，存在大量冗余代码和文件
解决措施：全面清理冗余代码和无效文件，将SQL脚本单独迁移至专用目录统一管理
7. 问题：前后端代码混合存放，边界不清晰，影响团队协作效率
解决措施：建立独立的frontend与backend子目录，实现前后端代码完全分离
2. 后端技术栈整体替换
1. 问题：原计划使用的MyBatis-Plus ORM框架未实际应用
解决措施：直接替换为Spring Data JPA，移除MyBatis-Plus相关依赖，基于JPA实现数据持久化
2. 问题：原计划使用的Spring Security安全框架未实际应用
解决措施：自定义实现JWT拦截器完成身份认证与权限控制，不引入Spring Security依赖
3. 问题：原计划引入的Redis缓存组件未实际应用
解决措施：聚焦核心业务需求，暂不引入非必要的缓存组件，简化技术架构
3. 后端开发实际问题
1. 问题：JWT拦截器异常处理逻辑存在缺陷，Token解析异常捕获机制失效
解决措施：重写JWT拦截器及全局异常处理器，完善异常捕获流程和响应逻辑
2. 问题：接口权限配置存在漏洞，拦截路径配置错误导致权限控制异常
解决措施：调整WebConfig中的拦截路径配置，修正拦截器注册路径模式，修复未授权访问漏洞
3. 问题：数据库初始表结构设计不完善，字段约束和数据格式需要优化
解决措施：逐步调整字段名和约束条件，重写数据库初始化脚本，重构字段设计及种子数据格式
4. 问题：个人信息修改核心业务逻辑实现不完善，存在功能缺陷
解决措施：重构个人信息修改的核心业务逻辑，优化代码结构，提升功能健壮性
4. 前端开发
1. 问题：原计划使用的Vue3前端技术栈未实际应用
解决措施：直接采用React技术栈进行前端开发，无过渡版本，快速完成前端项目初始化
2. 问题：前后端联调时Vite代理配置错误，导致前后端通信异常
解决措施：修正Vite代理配置，将代理目标调整为正确的后端服务地址，恢复前后端正常通信
5. 测试工作
1. 问题：测试工作整体滞后于开发进度
解决措施：在核心功能开发完成后立即开展手工测试，及时发现并记录系统缺陷
2. 问题：测试覆盖范围不完整，仅覆盖部分核心模块
解决措施：后续逐步扩大测试覆盖范围，补充未覆盖模块的测试用例
3. 问题：缺乏自动化测试体系，仅存在基础的Spring上下文加载测试
解决措施：计划引入自动化测试框架，逐步搭建完善的自动化测试体系
4. 问题：部分已知系统缺陷未及时修复
解决措施：安排专项开发时间，集中修复所有遗留缺陷，提升产品质量
6. Git协作
1. 问题：未执行计划的分支管理策略，仅使用单一主分支进行开发
解决措施：后续严格执行主分支+开发分支的分支策略，建立规范的分支管理体系
2. 问题：未按计划创建功能分支，所有开发工作直接在主分支进行
解决措施：后续所有功能开发均从开发分支拉出独立的功能分支，完成后再合并
3. 问题：缺乏代码审核流程，所有提交直接推送至主分支
解决措施：后续引入Pull Request代码审核机制，所有代码合并前必须经过审核
4. 问题：Git提交信息不规范，存在大量随意描述，可读性差
解决措施：后续严格遵守“[模块] 具体功能描述”的提交规范，提高提交信息的可读性
5. 问题：代码提交时间分布不均，存在集中爆发式提交
解决措施：合理规划开发进度，养成每日提交代码的习惯，避免集中式提交
6. 问题：多人并行开发时缺乏有效沟通，导致合并冲突和代码混乱
解决措施：加强团队内部实时沟通，提前同步开发计划，减少不必要的合并冲突
